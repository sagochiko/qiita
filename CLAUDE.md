# Qiita記事管理プロジェクト

## 注意
- ユーザーは音声入力を使用している。誤字・誤変換が含まれる可能性があるため、文脈から意図を汲み取ること。

## ルール
- 記事はQiita CLIで管理する（`public/`配下にフラットに配置）
- 画像は`images/<記事名>/`配下に配置（Qiitaアップロード後のURLを記事内で使用）
- サンプルコードは`samples/<記事名>/`配下に配置
- `images/`と`samples/`のディレクトリ名は記事ファイル名（拡張子なし）と揃えること
- 作業前にTODO.mdを確認・更新すること
- 学びはKNOWLEDGE.mdに追記すること

## 記事の書き方
- タイトルは具体的にする（例: ×「Lambdaについて」 ○「Lambda + DynamoDBでサーバーレスAPIを構築する」）
- ハンズオン形式を基本とする
- サンプルコードは動作確認済みのものを載せる
- AIは調査・構成・誤字検出の補助として使い、文章は自分の言葉で書く

## 文章スタイル
- 箇条書きと太字（**）を多用しない。自然な文章で書く
- 見出しにコロン（:）を使わない（例: ×「Lambda：概要」 ○「Lambdaの概要」）
- 抽象名詞の体言止めを避ける（例: ×「パフォーマンスの最適化」 ○「パフォーマンスを最適化する」）
- 絵文字を使わない
- AI生成っぽい文章にならないよう、自然な日本語を心がける

## Git規約

### ブランチ名
- `<type>/<短い説明>` の形式（例: `feat/lambda-article`, `fix/publish-workflow`）

### コミットメッセージ
- Conventional Commitsに従う: `<type>: <説明>`

| type | 用途 |
|---|---|
| feat | 新機能・新記事 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| chore | 設定・CI・依存関係など |
| refactor | リファクタリング |
| style | フォーマット変更（動作に影響なし） |
| test | テスト追加・修正 |

## 関連ファイル
- `KNOWLEDGE.md` - 記事執筆・運用の学び
- `TODO.md` - タスク管理
